01 — MOTIVATION

Why do we need multi‑threading in Java?

A single thread runs one instruction at a time. Real applications rarely wait for just one thing — they read files, respond to clicks, and talk to the network, often all at once.

01

Responsiveness

A UI thread stays interactive while a background thread loads data, instead of freezing until the work finishes.

02

Throughput

Modern CPUs ship with multiple cores. Threads let one program actually use more than one of them at a time.

03

Resource sharing

Threads in the same process share memory directly, which makes cooperating tasks cheaper than spinning up separate processes.

02 — IMPLEMENTATION

How to implement a thread

Java gives you two doors into the same room: extend Thread, or implement Runnable. The second is generally preferred, since Java only allows single inheritance and a class implementing Runnable is free to extend something else.

Option A — extend Thread

Override run(), then call start() — never run() directly, or it just executes like a normal method call on the current thread.

class Worker extends Thread {
    public void run() {
        System.out.println("Running in: " + getName());
    }
}

Worker w = new Worker();
w.start(); // schedules the thread

Option B — implement Runnable

Pass the task to a Thread instead of becoming one. This keeps the "what to do" separate from "how it runs".

class Task implements Runnable {
    public void run() {
        System.out.println("Task on: "
            + Thread.currentThread().getName());
    }
}

new Thread(new Task()).start();

Full example — two threads counting concurrently

public class CounterDemo {
    public static void main(String[] args) {
        Runnable counter = () -> {
            for (int i = 1; i <= 5; i++) {
                System.out.println(Thread.currentThread().getName() + " -> " + i);
                try { Thread.sleep(200); } catch (InterruptedException e) {}
            }
        };

        new Thread(counter, "Thread-A").start();
        new Thread(counter, "Thread-B").start();
    }
}
// Output interleaves unpredictably — that unpredictability is the point.
03 — SYNCHRONIZATION

The bank account withdrawal problem

Two threads withdraw from the same account at nearly the same moment. Both read the same starting balance before either writes back the result — the classic race condition.

race condition

Balance starts at 1000. Two withdrawals of 800 run "at the same time." Neither should succeed alone, yet without protection both can go through, sending the balance negative.

class BankAccount {
    private double balance = 1000;

    public void withdraw(double amount) {
        if (balance >= amount) {
            // another thread can slip in right here, between the check and the update
            try { Thread.sleep(50); } catch (Exception e) {}
            balance -= amount;
            System.out.println("Withdrew " + amount + ", balance = " + balance);
        } else {
            System.out.println("Insufficient funds");
        }
    }
}
// Both threads pass the "if" check before either subtracts -> balance ends up wrong.
synchronized

The synchronized keyword locks the method on the object's monitor. Only one thread may execute it at a time, so the check‑then‑act sequence can no longer be interrupted.

class BankAccount {
    private double balance = 1000;

    public synchronized void withdraw(double amount) {
        if (balance >= amount) {
            try { Thread.sleep(50); } catch (Exception e) {}
            balance -= amount;
            System.out.println("Withdrew " + amount + ", balance = " + balance);
        } else {
            System.out.println("Insufficient funds");
        }
    }
}
// The second thread now waits for the lock before it can even read the balance.
04 — THREAD LIFE CYCLE

Five states, in order

Every Java thread moves through this sequence — sometimes cycling between Runnable and Blocked/Waiting many times before it finally terminates.

STATE 1

New

Created with new Thread() but start() hasn't been called yet.

STATE 2

Runnable

Eligible to run; waiting for the OS scheduler to give it CPU time.

STATE 3

Running

Actually executing run() on the processor right now.

STATE 4

Blocked / Waiting

Paused for a lock, sleep(), wait(), or I/O to finish.

STATE 5

Terminated

run() has finished, or the thread exited abnormally.

05 — NECESSARY THINGS & PROJECT

What the project needs to demonstrate

The companion repository turns every concept above into runnable code. Before checking it out, make sure the environment covers this checklist.

  • JDK 8 or newer installed and on the PATH, verified with java -version.
  • An IDE or editor comfortable with multi-class Java projects (IntelliJ IDEA, Eclipse, or VS Code with the Java extension pack).
  • A working example of both thread creation styles: extending Thread and implementing Runnable.
  • A deliberately broken, unsynchronized bank withdrawal demo, followed by a fixed version using synchronized.
  • Console logging that prints the current thread's name at each step, so interleaving is visible when the program runs.
  • A short write-up mapping observed behavior back to the five thread-lifecycle states.
